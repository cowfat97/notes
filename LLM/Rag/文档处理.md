# RAG 文档处理

## What — 是什么

文档处理是 RAG 的第一道关卡。把任意格式的文件（PDF、Word、Markdown、PPT）转成结构化的文本块，供下游向量化和检索使用。

核心四步：

```text
任意格式文件(.pdf/.docx/.md/.pptx)
  → 布局识别（区分标题/正文/表格/图片）
  → 元素提取（表格保留行列，图片走 OCR，文字保留层级）
  → 转为 Markdown（统一结构格式）
  → 分割（结构感知切分 + 父子块）
```

## Why — 为什么

文档处理不好，Embedding 再准、Reranker 再强都白费。

- PDF 没有 `<h1>` `<table>` 标签，只有"在 x,y 坐标画了某段文字"。不识别布局，标题跟正文混在一起，表格变成散落文字。
- OCR 选错引擎（Tesseract 做中文），"线程池"变成"线裎池"，后续全链路污染。
- 切分时按固定长度拦腰切断，标题和正文分到两个 chunk，语义碎片化。

**一句话**：文档解析质量 = 检索质量的上限。解析丢了 30% 的结构，后面的 Embedding 再好也只能用 70% 的信息。

## How — 怎么做

### 1. 布局识别

`unstructured` 负责找出页面上哪块是标题、哪块是正文、哪块是表格。

| strategy | 原理 | 速度 | 精度 |
|----------|------|------|------|
| `auto` | 有文字层直接提取，没有才 OCR | 快 | 中 |
| `hi_res` | YOLOX 目标检测模型先识布局 | 慢 | 高 |
| `ocr_only` | 整页 OCR，跳过布局模型 | 慢 | 低 |

`hi_res` 技术原理：YOLOX CSPDarkNet 提取特征 → FPN 多尺度融合 → 预测每个 bbox 坐标 + 类别。模型训练时学了：14pt 加粗居中 = 标题，10pt 对齐多行 = 正文。

```python
from unstructured.partition.auto import partition

elements = partition(filename="doc.pdf", strategy="hi_res")
for el in elements:
    print(el.category)  # Title / NarrativeText / Table / Image / ListItem
```

### 2. 元素提取（图片 + 表格 + OCR）

**OCR 选型**：PaddleOCR。中文精度和表格识别都比 Tesseract 强。

**unstructured + PaddleOCR 连接**：`unstructured` 管"哪块是文字"，PaddleOCR 管"那块是什么字"。装 `unstructured[paddleocr]` 后内部自动对接。

```python
from unstructured.partition.pdf import partition_pdf

# 普通文档
elements = partition_pdf("doc.pdf", strategy="ocr_only", ocr_languages="chi_sim")

# 银行/手写/发票 — 精细控制
from paddleocr import PaddleOCR
from unstructured_paddle import PaddleOCRAdapter

ocr = PaddleOCR(lang="ch", use_angle_cls=True, use_gpu=False, det_db_thresh=0.2)
elements = partition_pdf("doc.pdf", strategy="ocr_only", ocr=PaddleOCRAdapter(ocr=ocr))
```

**PaddleOCR 内部**：DB（圈出文字区域）→ Cls（判断横排竖排）→ CRNN（CNN 视觉特征 + BiLSTM 序列建模 + 对齐）。

**场景调参**：

| 场景 | `det_db_thresh` | `use_angle_cls` |
|------|----------------|-----------------|
| 普通文档 | 0.3 | 可关 |
| 银行申请件 | 0.2（淡字也要抓） | 开 |

### 3. 转为 Markdown

不同格式经过前两步处理后，统一转为 Markdown——保留 `##` 标题、`|表格|`、` ```code``` `。

```python
def elements_to_markdown(elements):
    md_parts = []
    for el in elements:
        if el.category == "Title":
            md_parts.append(f"## {el.text}\n")
        elif el.category == "Table":
            md_parts.append(html_to_md(el.metadata.text_as_html))
        elif el.category == "Image":
            md_parts.append(f"[图片: {ocr(el.image)}]\n")
        elif el.category == "ListItem":
            md_parts.append(f"- {el.text}\n")
        else:
            md_parts.append(f"{el.text}\n\n")
    return "\n".join(md_parts)
```

**为什么是 Markdown**：LLM 原生理解，MarkdownTextSplitter 原生理解（不横切表格和代码块），跟笔记格式一致。

### 4. 分割

**核心矛盾**：检索要小块（精准匹配 query），LLM 要大块（完整上下文）。

切分方式从差到好分五层。

#### 4.1 固定长度切分

```python
chunks = [text[i:i+300] for i in range(0, len(text), 300)]
```

三个问题：单词拦腰切断、标题被塞进前一个 chunk 的结尾、语义不相关的内容被硬粘在一起。**永远不要用。**

#### 4.2 标点边界切分

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=300, chunk_overlap=50,
    separators=["\n\n", "\n", "。", "，", " "]
)
```

`separators` 是优先级列表：先试段落边界，不行逐级下沉到句子、短句、最后空格。句子保持完整，不会在单词中间切断。**局限**：对所有文档一视同仁，不感知 Markdown 结构。

#### 4.3 Markdown 结构感知切分

```python
from langchain_text_splitters import MarkdownTextSplitter

splitter = MarkdownTextSplitter(chunk_size=1200, chunk_overlap=100)
```

| 结构 | 策略 |
|------|------|
| `##` / `###` 标题 | 优先在此处切——新标题 = 新主题 |
| 表格 | 绝对不切——行列结构跨行就废了 |
| 代码块 | 绝对不切——语法跨 chunk 全毁 |
| `- 列表项` | 列表项之间可以切 |
| 普通段落 | 按 `\n\n` → `\n` → `。` 优先级 |

#### 4.4 中文递归切分

课程自定义 `ChineseRecursiveTextSplitter`，继承 `RecursiveCharacterTextSplitter`，换中文标点优先级：

```python
separators = [
    "\n\n",           # 段落
    "\n",             # 换行
    "。|！|？",      # 中文句号
    "\.\s|\!\s|\?\s", # 英文句号
    "；|;\s",         # 分号
    "，|,\s"          # 逗号
]
```

原版没有中文标点，不加的话中文标点边界切分完全失效。

**按格式选择切分器**：

```python
is_markdown = (file_extension == ".md")
parent_splitter = MarkdownTextSplitter(chunk_size=1200) if is_markdown \
    else ChineseRecursiveTextSplitter(chunk_size=1200)
```

#### 4.5 父子块

**核心矛盾**的答案：子块搜，父块看。

```
父块 1200 token（给 LLM 看 = 完整章节上下文）
  ├─ 子块1 300 token ← 检索命中 → 返回整个父块
  ├─ 子块2 300 token
  ├─ 子块3 300 token
  └─ 子块4 300 token
```

每个子块携带父块信息：

```python
sub_chunk.metadata["parent_id"] = parent_id          # 指向父块
sub_chunk.metadata["parent_content"] = parent_doc.page_content  # 父块全文
sub_chunk.metadata["id"] = f"{parent_id}_child_{k}"  # 子块唯一 ID
```

检索链路：query 向量化 → 命中子块 → 不返回子块内容 → 返回 parent_content → LLM 看到完整上下文。

#### 4.6 辅助机制

**Overlap**：前后 chunk 重叠 100-150 token，避免关键信息恰好落在切断点。

**Metadata 拼接**：每个 chunk 拼上标题路径：

```
原始: "corePoolSize 控制核心线程数..."
增强: "Java并发 > 线程池 > 核心参数\n\ncorePoolSize 控制核心线程数..."
```

检索时"线程池"和"核心线程数"同时存在在向量中，命中率更高。

**chunk_size 取值**：父块 1000-1200 token，子块 200-300 token，overlap 100-150 token。

#### 4.7 完整切分流程

```python
def process_documents(directory_path):
    documents = load_documents_from_directory(directory_path)  # 1. 加载

    # 2. 准备切分器（通用 + Markdown 专用）
    parent_splitter = ChineseRecursiveTextSplitter(chunk_size=1200, chunk_overlap=100)
    child_splitter  = ChineseRecursiveTextSplitter(chunk_size=300,  chunk_overlap=50)
    md_parent_splitter = MarkdownTextSplitter(chunk_size=1200, chunk_overlap=100)
    md_child_splitter  = MarkdownTextSplitter(chunk_size=300,  chunk_overlap=50)

    child_chunks = []
    for i, doc in enumerate(documents):
        # 3. 按格式选切分器
        is_md = doc.metadata["file_path"].endswith(".md")
        p_splitter = md_parent_splitter if is_md else parent_splitter
        c_splitter = md_child_splitter  if is_md else child_splitter

        # 4. 父块切分 → 5. 子块切分 → 6. 子块继承父块信息
        parent_docs = p_splitter.split_documents([doc])
        for j, parent_doc in enumerate(parent_docs):
            parent_id = f"doc_{i}_parent_{j}"
            parent_doc.metadata["parent_id"] = parent_id
            parent_doc.metadata["parent_content"] = parent_doc.page_content

            sub_chunks = c_splitter.split_documents([parent_doc])
            for k, sub_chunk in enumerate(sub_chunks):
                sub_chunk.metadata.update({
                    "parent_id": parent_id,
                    "parent_content": parent_doc.page_content,
                    "id": f"{parent_id}_child_{k}"
                })
                child_chunks.append(sub_chunk)

    return child_chunks  # 存入 Milvus
```

#### 4.8 语义分割模型

`AliTextSplitter`（达摩院 BERT 文档语义分割模型）让模型判断"这个位置是不是语义边界"。

| 场景 | 是否需要 |
|------|---------|
| Markdown 笔记（有 `##` 标题） | 不需要，标题就是边界 |
| PDF OCR 纯文本 | **需要** |
| 合同、法规（格式被 OCR 吞掉） | **需要** |

笔记场景 `MarkdownTextSplitter` 已足够，银行文档场景语义分割模型有意义。

## 技术栈

| 环节 | 工具 |
|------|------|
| 布局识别 | `unstructured` + YOLOX |
| OCR | `PaddleOCR` (DB + CRNN) |
| 表格提取 | PP-TableStructure / `pdfplumber` |
| 格式统一 | 自拼装 + `markdownify` |
| 结构切分 | `MarkdownTextSplitter` + `ChineseRecursiveTextSplitter` |
| 父子块 | 代码逻辑 |
| 语义分割 | `AliTextSplitter`（可选） |
